# AdverseIQ — Complete Features List
**As of March 2026 | Backend implementation**

---

## Design Principles

- **K2 is the source of truth** for all clinical outputs — urgency, confidence, recommendation, safe alternative
- **Code-side logic exists only to catch K2 failures** — escalate urgency if K2 underestimates, fill gaps if K2 returns nothing, never replace K2's judgment
- **Two endpoints, one API key** — standard endpoint for fast strategies, agentic endpoint for deep investigation
- **Demo fallbacks are last resort** — only triggered if K2 fails and JSON repair fails; never used as a first-choice data source

## Analysis Strategies

### Rapid Check
Single drug-pair confirmation against a pre-loaded interaction record.

- K2 confirms or refutes whether the known interaction explains the symptom
- `recentlyAdded` injects a `NOTE:` that focuses K2 on the newly introduced drug
- Safe alternative generated by K2 if interaction confirmed; static fallback if K2 call fails
- Urgency comes from K2; rule-based assessor only escalates, never downgrades

### Mechanism Trace
Full pharmacological causal chain from drug action to observed symptom.

- K2 constructs a step-by-step causal chain with evidence citations per step
- Each step includes mechanism, expected finding, evidence string, and source tag (`database|literature|mechanism`)
- `recentlyAdded` injects a `CRITICAL CONTEXT:` block anchoring the chain to the newly changed drug
- K2's own `confidence_factors` array returned alongside engine annotations
- Safe alternative generated by K2 for the primary interaction pair

### Mystery Solver (Agentic)
Multi-hypothesis investigation — K2 drives its own evidence gathering via tool calls.

- K2 receives the patient case and investigates autonomously over multiple turns
- Generates all plausible hypotheses: drug-drug, drug-herb, adverse effect, medication failure, disease progression, non-adherence, withdrawal
- Ranks hypotheses by confidence; explains why each rejected hypothesis was eliminated
- `recentlyAdded` injects a `CRITICAL — DIFFERENTIAL FOCUS:` block restructuring investigation priority around the changed drug
- Available as non-streaming (`POST /api/analyze`) and streaming SSE (`POST /api/analyze/stream`)

## Agentic Capabilities (Mystery Solver only)

K2 calls these tools autonomously during its investigation. You execute them; K2 reasons on the results.

| Tool | What K2 uses it for |
|---|---|
| `lookup_drug_interaction(drug_a, drug_b)` | Checks every drug pair against the interactions database |
| `search_pubmed(query)` | Finds recent case reports and studies for pairs not in the database |
| `get_drug_class(drug_name)` | Assesses serotonergic load, QT burden, bleeding risk, CYP inducer/inhibitor status |
| `get_cyp_profile(drug_name)` | Verifies CYP enzyme inhibition, induction, and substrate relationships |
| `get_safe_alternative(drug_to_replace, indication, interacting_drug)` | Suggests a therapeutically equivalent safer substitute after identifying the primary interaction |

**Parallel tool calls** — K2 may call multiple tools in a single turn (e.g. `lookup_drug_interaction` and `get_drug_class` simultaneously). All are executed and returned before K2 continues.

**Agent loop** — runs up to **5 turns** (`MAX_TURNS = 5`). Tool-call turns use the agentic endpoint with tools array; the final answer turn switches to JSON mode + logprobs, with tools array **omitted entirely** (`tool_choice: "none"` is broken on build-api — omitting the array is the correct pattern).

**`tools_used`** — list of tool names K2 called is returned in the AnalysisResult. Displayed in the frontend Investigation tab.

## Confidence Engine

K2 is the source of truth for the confidence score. The engine **passes K2's score through unchanged** — it does not blend it with rule-based factor weights (the previous 70/30 blend was removed; it was adding a near-constant offset using hardcoded `True` flags, which was false precision).

### Score
- Comes directly from K2's output
- On Mystery Solver (non-streaming): logprob calibration may discount it **before** it reaches the confidence engine
- On Rapid Check and Mechanism Trace: K2's raw score, passed through unchanged
- The engine generates **annotations only** — it does not apply point weights or modify the numeric score

### Logprob Calibration (Mystery Solver non-streaming only)
Measures how certain the model actually was when it wrote the confidence number.

| Model certainty | Action |
|---|---|
| ≥ 0.7 (high) | Score kept unchanged |
| 0.3–0.7 (medium) | Score discounted 10% |
| < 0.3 (low) | Score discounted 20% |
| Logprobs unavailable | Score kept unchanged |

The calibration result appears as an annotation in `confidence_factors` — visible to judges in the UI.

### Annotations
Human-readable labels that explain why the score is what it is. Generated from evidence-presence flags. Do not affect the number.

**Positive annotations** (shown when true):
- Known interaction found in database
- Mechanism is pharmacologically plausible
- Reported symptom matches expected pharmacological effect
- Symptom onset timing correlates with drug introduction
- PubMed literature supports this interaction
- Multiple independent mechanisms converge on this conclusion

**Negative annotations** (shown when true):
- Key expected clinical signs are absent
- An equally plausible alternative hypothesis exists
- Drug dose may be insufficient to produce this interaction

## Urgency Resolution

K2 is the primary source. The rule-based assessor runs independently and can only escalate.

```
K2: urgent  + Assessor: emergent  →  emergent   (escalation)
K2: emergent + Assessor: routine  →  emergent   (K2 kept, never downgraded)
K2: urgent  + Assessor: routine   →  urgent     (K2 kept)
K2: invalid + Assessor: urgent    →  urgent     (assessor as pure fallback)
```

`urgency_reason` comes from K2 directly. Escalation reason shown only if the assessor overrode K2.

**Escalation patterns** (rule-based, fire only when K2 underestimates):
- 2+ serotonergic drugs + 2+ serotonin syndrome symptoms → `emergent`
- 2+ QT-prolonging drugs + cardiac symptoms → `urgent`
- Anticoagulant + antiplatelet + bleeding symptoms → `urgent`
- Statin + fibrate or macrolide + myopathy symptoms → `urgent`

## Safe Alternative

K2 generates the suggestion. Static lookup is code fallback only if the K2 call fails.

- **Rapid Check**: fired if `interaction_found` is true; K2 call with `temperature=0` for determinism
- **Mechanism Trace**: fired if a primary interaction record exists in the database
- **Mystery Solver**: K2 calls `get_safe_alternative` tool autonomously during its investigation
- Result returned as `safe_alternative` string in AnalysisResult
- If K2 call fails: static fallback table covers the three demo pairs only

## `recentlyAdded` Differential Analysis

Optional field on the request. When provided, all three strategies restructure their prompt.

| Strategy | Injection style | Effect |
|---|---|---|
| Rapid Check | `NOTE:` | Soft focus on whether the new drug explains the symptom |
| Mechanism Trace | `CRITICAL CONTEXT:` | Anchors causal chain to the newly introduced drug |
| Mystery Solver | `CRITICAL — DIFFERENTIAL FOCUS:` | Restructures K2's investigation order; new drug pairs checked first; temporal correlation weighted as strong evidence |

## Streaming (Mystery Solver)

SSE event stream from `POST /api/analyze/stream`.

| Event | Data | When |
|---|---|---|
| `stage` | Progress label string | Pipeline milestones and tool call labels |
| `thinking` | K2 reasoning token | During K2's reasoning stream |
| `tool_summary` | Summary of tools K2 called | After agent loop completes |
| `result` | Complete AnalysisResult dict | Final assembled result |
| `error` | Error message string | On any failure |

**Note:** logprob calibration is not applied on the streaming path (logprobs only available on non-streaming final turn). Use `POST /api/analyze` for the fully calibrated result.

## JSON Repair Pipeline (Standard Endpoint)

Used by Rapid Check and Mechanism Trace. Applied in order — stops at first success:

1. Strip `<think>...</think>` tags from response
2. Extract JSON from markdown code fences
3. Stack-match `{...}` braces to isolate the first complete object
4. Repair trailing commas before `}` or `]`
5. Close unclosed `[` and `{` structures
6. If parse still fails — **self-repair call**: K2 is asked to fix its own malformed output (second API call to standard endpoint)
7. If self-repair fails — use `demo_fallback` if provided, otherwise raise

## AnalysisResult TypeScript Interface

Complete shape returned by all three strategies:

```typescript
interface AnalysisResult {
  strategy: "rapid" | "mechanism" | "hypothesis";
  urgency: "routine" | "urgent" | "emergent";
  urgency_reason: string;
  overall_confidence: number;           // 0–100; K2's score, logprob-calibrated on Mystery Solver
  confidence_factors: {                 // Annotation labels for the UI
    factor: string;
    direction: "increases" | "decreases";
    weight?: "high" | "medium" | "low";
  }[];
  recommendation: string;
  disclaimer: string;

  // Rapid Check + Mechanism Trace
  interaction_found?: boolean;          // Rapid Check only
  mechanism?: string;                   // Rapid Check only
  causal_steps?: {                      // Both Rapid Check (1 step) and Mechanism Trace
    step: number;
    mechanism: string;
    expected_finding: string;
    evidence: string;
    source: "database" | "literature" | "mechanism";
  }[];
  db_interaction?: object | null;       // First matching DB record; null on Mystery Solver

  // Mystery Solver only
  hypotheses?: {
    id: string;                         // e.g. "H1", "H2"
    description: string;
    mechanism: string;
    confidence: number;
    supporting_evidence: string[];
    rejecting_evidence: string[];
    status: "supported" | "possible" | "rejected";
    evidence_source: "database" | "literature" | "mechanism";
    pubmed_refs: string[];              // PMIDs
  }[];
  top_hypothesis?: string;             // e.g. "H1"
  tree_nodes?: object[];               // React Flow nodes for hypothesis tree
  tree_edges?: object[];               // React Flow edges for hypothesis tree
  // Note: hypotheses with confidence < 15 are pruned from the tree (but still in `hypotheses`)

  // All strategies
  safe_alternative?: string;
  // K2-generated safer substitute suggestion.
  // Display in RecommendationCard in a green-tinted section labelled
  // "Safer Alternative Considered".

  tools_used?: string[];
  // Mystery Solver only. Tool names K2 called, in call order.
  // e.g. ["lookup_drug_interaction", "get_drug_class", "search_pubmed", "get_safe_alternative"]
  // Display in EvidencePanel as "Investigation" tab.
}
```

Request shape:

```typescript
interface AnalysisRequest {
  medications: { displayName: string; [key: string]: any }[];
  symptoms: { description: string; [key: string]: any }[];
  patientContext?: object;
  strategy: "rapid" | "mechanism" | "hypothesis";

  recentlyAdded?: string;
  // Generic name of a drug recently added or changed.
  // When provided, all three strategies focus their analysis on this drug.
  // Surface as a UI toggle on the medication card:
  // "This medication was recently added or changed"
}
```

### Hypothesis Tree (Mystery Solver)

`tree_nodes` and `tree_edges` are pre-built React Flow graph data returned in the Mystery Solver result.

- Root node type: `patientNode` — represents the patient case
- Hypothesis node types: `hypothesisNode` (supported/possible) or `rejectedNode`
- Mechanism node type: `mechanismNode` — supporting mechanism steps
- Node color: green (`#16A34A`) ≥ 70%, amber (`#D97706`) ≥ 40%, gray < 40% or rejected
- **Pruning**: hypotheses with `confidence < 15` are excluded from the tree nodes/edges (but remain in the `hypotheses` array)
